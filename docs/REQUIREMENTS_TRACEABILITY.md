# Challenge Requirements Traceability

| PDF point | Product behavior | Acceptance proof |
|---|---|---|
| Pick one phone-priced vertical | Laboratory-equipment service | Product copy and single schema |
| Voice interview | ElevenLabs intake agent | Live recorded intake |
| Document intake | Synthetic service report/quote | Upload and extracted evidence |
| Same structured spec | `ServiceScope` | Schema validation test |
| User confirms spec | Confirmation gate | Correction updates canonical scope |
| Three distinct styles | OEM, lowballer, stonewaller | Three live sessions |
| Describe job consistently | Scope hash | Same hash on all call records |
| Handle friction | Interrupt/refusal/AI question policies | Simulation tests |
| Itemized comparable quotes | `ServiceQuote` | Deal Room matrix |
| Show call-list origin | Tavily discovery or curated demo directory | Dated provider-source panel |
| Negotiate with leverage | `checkLeverage` tool | Verified competing quote event |
| Price/terms change | `Concession` | Before/after timeline |
| 30% low quote is a warning | Deterministic outlier rule | Red-flag fixture test |
| Recordings and transcripts | Post-call webhooks | Clickable evidence |
| Explain recommendation | Award Memo | Reason and assumptions visible |
| Disclose AI | Opening line and response policy | Test transcript |
| Never bluff | Honesty Firewall | Blocked unsupported claim |
| Structured ending | Quote/callback/decline | Status on every call |

## Strong submission alignment

- Real negotiation: private counterparty concession policy, not scripted dialogue.
- Provable pain: one instrument and comparable service scope.
- Closed loop: intake through award memo.
- Conversation engineering: friction and tool-use tests.
- Honest failure handling: incomplete outcomes remain visible.

## Weak submission protections

- No screenplay: randomized bounded counterparty term revealed after the call.
- No incomparable quotes: scope-match and exclusions gate ranking.
- No polished fragment: all four screens share one state model.
- No stack theater: each service maps to an observable requirement.
- No bluffing: leverage validation is enforced outside the prompt.
